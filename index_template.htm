External Software

January 21, 2016

All external software that's built internally must follow the following naming scheme:

PACKGE_NAME-SOURCE_RELEASE_VERSION-RPM_VERSION.rp[OS_Base_Release].ARCH.rpm*

In this scheme we can expect our internal Python-2.7 package for x64 CentOS-5.x that contains the 2.7.5 upstream release to look like so:

python27-2.7.5-1.rp5.x86_64.rpm

This naming scheme provides us with several powerful pieces of information: The release number of the upstream code we are packaging (2.7.5 in our case) That this software is build by the Rubicon Project and is targeted for CentOS-5.x (as per the .rp5. bit) The code is compiled for x86_64 architecture If we were to recompile the following code, using the same upstream release we would then append a "1" after rp5. This designates that we used the same source, but modified how we either compiled the code or restructured files contained in the RPM CPIO archive. For example the first update to our Python-2.7 package would look like so:

python27-2.7.5-1.rp5.1.x86_64.rpm *

This naming scheme for versioning and release numbers should work in most cases, however sometimes when adding the .rp5 flag to a package that formerly did not may not be parsed correctly by RPM. If this happens, it is best to set an "Epoch: [number]" before the Version and Release numbers. More information on spec file versioning bestpractices can be found here: http://fedoraproject.org/wiki/Packaging:NamingGuidelines# Package_Versioning
